## 1. 목표

이 페이지의 목적은 단순 문서 렌더링이 아니라, 방문자가 짧은 시간 안에 아래를 이해하도록 만드는 것이다.

- 이 확장앱이 무엇인지
- 어떤 문제를 해결하는지
- 다른 AI 확장 기능과 무엇이 다른지
- 현재 구현된 기능과 앞으로 확장될 방향이 무엇인지
- 누구에게 적합한 제품인지

즉, README나 기술 문서가 아니라 **제품 소개 + 신뢰 형성 + 관심 유도**에 초점을 둔다.

## 2. 페이지 성격

### 2.1 페이지 타입

- 제품 소개용 웹 랜딩 페이지
- 문서형보다 제품형 구조 우선
- 기술 세부보다 사용 시나리오와 가치 전달 우선

### 2.2 목표 독자

우선 순위:

1. 확장앱을 처음 보는 일반 사용자
2. 생산성 도구를 찾는 업무 사용자
3. 브라우저 자동화나 AI 코파일럿에 관심 있는 파워 유저
4. 추후 도입 가능성을 보는 팀/조직 사용자 *(내부 도구로서의 도입)*
5. **해당 확장앱 또는 확장앱에 사용된 기술을 이용하여 사업에 적용하고 싶은 기업** *(기술 내재화 · OEM · 파트너십 · 레퍼런스 확보)*

> **4번과 5번의 차이**: 4번은 "팀의 생산성 도구로 이 제품을 구매/도입"하는 관점이고, 5번은 "이 제품이 사용하는 기술 스택(페이지 문맥 기반 AI, Action Tools 에이전트, 스크립트 실행/등록 인프라 등)을 자사 서비스에 적용"하려는 관점이다. 전자는 제품 사용자, 후자는 기술 파트너/도입사이므로 메시지와 전환 경로가 달라야 한다 (§5.10 참조).

### 2.3 기대 행동

방문자가 페이지를 보고 다음 행동 중 하나를 하게 만드는 것이 목표다.

**일반/업무/파워 유저 (#1~#3)**
- 제품에 대한 전반 이해
- 설치 또는 체험 의사 형성
- 기능 확인
- 향후 Studio / Floating Helper / Script workflow에 대한 기대 형성

**팀/조직 사용자 (#4)**
- 조직 내 도입 가능성 검토
- 안전/운영 원칙에 대한 신뢰 형성

**기술 적용 관심 기업 (#5)** *(신규)*
- 기술 스택과 확장 가능성 이해
- 파트너십/기술 내재화 가능성에 대한 관심 형성
- 문의 경로 확보 (이메일/폼 등)

> 1차 구현에서 #5의 전환 경로는 최소 수준(문의 이메일 링크)으로 충분하다. 실제 세일즈 파이프라인은 후속 이터레이션에서 다룬다.

## 3. 핵심 메시지

랜딩 페이지 전체에서 일관되게 전달해야 할 메시지는 다음이다.

### 3.1 메인 메시지

**Web AI Assistant는 웹페이지 문맥을 이해하고, 질문에 답하고, 필요하면 브라우저를 직접 조작하는 AI 기반 브라우저 생산성 도구다.**

### 3.2 보조 메시지

- 단순 챗봇이 아니라 페이지를 읽고 행동하는 코파일럿
- 선택 텍스트, 페이지 구조, 입력 필드, 스크립트까지 다루는 확장앱
- 단발성 자동화에서 재사용 가능한 사용자 맞춤 확장으로 진화 중
- **(B2B 보조 메시지)** 페이지 문맥 이해 · Action Tools 에이전트 · 스크립트 실행 인프라는 자사 서비스에 적용할 수 있는 **재사용 가능한 기술 기반**이다 *(5.10 섹션에서 구체화)*

### 3.3 피해야 할 메시지

- 완전 무인 자동화 도구처럼 과장된 표현
- 민감한 사이트에서 무조건 안전하다는 식의 표현
- 아직 미구현된 기능을 현재 완성 기능처럼 표현하는 문장

## 4. 페이지 구성 원칙

### 4.1 구조 원칙

- 위에서 아래로 읽을수록 제품 이해가 깊어지는 구조
- 첫 화면은 “무엇을 하는가”에 집중
- 중간은 “어떻게 도움이 되는가”와 “무엇이 다른가”
- 하단은 “확장 방향”과 “신뢰/안전” 정리

### 4.2 콘텐츠 원칙

- 한 섹션당 메시지는 하나만
- 기능 나열보다 사용 장면 중심
- 구현됨 / 진행 중 / 기획 단계 구분 명확히

### 4.3 시각 원칙

- 기술 문서 느낌보다 제품 소개 느낌
- 스크린샷이나 흐름 카드가 들어가도 자연스럽게 붙는 구조
- 너무 많은 표보다 카드/섹션/짧은 설명 중심

## 5. 권장 정보 구조

아래 순서를 기본 골격으로 한다.

### 5.1 Hero

첫 화면에서 전달할 내용:

- 제품명
- 한 줄 설명
- 핵심 키 메시지 2~3개
- 대표 CTA

예시 메시지 방향:

- 웹페이지를 이해하고, 질문하고, 자동화하는 AI 코파일럿
- 브라우저 안에서 바로 쓰는 AI 채팅 + 액션 + 스크립트 워크플로우

권장 CTA:

- 기능 살펴보기
- 사용 시나리오 보기
- 설치/체험하기

### 5.2 문제 정의

사용자가 기존에 겪는 문제를 짧게 보여준다.

예:

- 페이지 내용을 읽고 다시 정리해야 함
- 웹 작업이 반복적이고 수동적임
- AI는 답변만 하고 실제 작업은 사용자가 다시 해야 함
- 사이트별 자동화는 코드 작성 장벽이 높음

### 5.3 해결 방식

이 확장앱이 문제를 어떻게 푸는지 3축으로 정리한다.

- 페이지 문맥 기반 AI
- 브라우저 액션 자동화
- 스크립트 기반 확장성

이 섹션은 소개서의 2장 내용을 가장 압축적으로 재구성하는 핵심 섹션이다.

### 5.4 주요 기능

기능 카드 섹션으로 구성한다.

후보 카드:

- AI 채팅
- 페이지 도우미
- 텍스트 선택 기반 기능
- 문서/폼 자동입력
- Action Tools 브라우저 에이전트
- 이미지/스크린샷
- 스크립트 실행 및 등록

각 카드는:

- 기능명
- 한 줄 가치
- 대표 사용 예시

정도로 제한한다.

### 5.5 사용 시나리오

방문자가 실제로 어떻게 쓸지 보여준다.

권장 시나리오:

1. 기사 문장 선택 후 번역/질문
2. 그룹웨어 문서 작성 보조
3. 탭 정리 및 페이지 탐색 자동화
4. 사이트 다크모드/광고 숨김 적용

이 부분은 단순 기능보다 제품 이해를 빠르게 만든다.

### 5.6 차별화 포인트

비교형 메시지로 정리한다.

- 답변형 AI vs 행동형 AI
- 단발성 자동화 vs 재사용 가능한 확장 자산
- 텍스트 중심 AI vs 페이지 구조/필드/스크립트까지 다루는 AI

### 5.7 지원 AI 모드

현재 지원 상태를 짧게 보여준다.

- OpenAI
- LM Studio
- Didim
- Gemini
- Ollama는 “검토 중” 또는 “준비 중”으로 분리

이 섹션은 기술적이지만 사용자가 확장앱의 유연성을 이해하는 데 중요하다.

### 5.8 안전 및 운영 원칙

민감 작업에 대한 보수적 태도를 명확히 표현한다.

핵심 메시지:

- 자동화는 가능하지만 승인/확인 경계가 중요함
- 영구 스크립트는 명시적 흐름으로 관리
- 고위험 제출/인증은 사용자 확인 중심

이 섹션은 신뢰 형성용이다.

### 5.9 확장 방향

현재 진행 중인 방향을 제품 로드맵 느낌으로 짧게 소개한다.

- Floating Helper
- Session Script Continuity
- Extension App Studio

주의:

- “구현됨”이 아니라 “확장 방향”으로 표현
- 과장 없이 현재 상태 구분

### 5.10 비즈니스 적용 / 기술 파트너십 *(신규 — B2B 대상)*

> **대상**: §2.2 #5 (해당 확장앱 또는 확장앱에 사용된 기술을 이용하여 사업에 적용하고 싶은 기업)
> **위치**: Roadmap 섹션 뒤, 마무리 CTA 앞. 바닥까지 스크롤한 고관여 방문자가 자연스럽게 만나는 위치.

일반 사용자용 CTA로 끝나기 전에, **기술 재사용과 파트너십 가능성**을 가진 방문자를 위한 독립 섹션을 둔다. 일반 방문자에게는 "내 얘기가 아님"이 즉시 인식되도록 배경·타이포가 약간 다르게 처리되고, B2B 방문자에게는 "여기가 우리 대화가 시작되는 지점"임이 드러나야 한다.

#### 메시지 방향

- 이 확장앱이 사용하는 기술은 단일 제품에만 묶여 있지 않다
- 페이지 문맥 이해 · Action Tools 에이전트 · 스크립트 실행/등록 인프라는 **다른 서비스에도 적용 가능한 재사용 가능 기반**이다
- 도입 방식은 제품 그대로 OEM, 기술 내재화, 공동 개발 등 여러 층이 있을 수 있다
- 과장 없이 "가능한 것"과 "아직 논의 단계인 것"을 구분한다

#### 구성 요소

1. **Eyebrow 라벨**: "For Businesses" / "기술 파트너십"
2. **H2**: 한 줄 가치 제안 (예: "이 확장앱의 기술을 자사 서비스에 적용할 수 있습니다")
3. **Sub**: 2~3줄 설명 (위 메시지 방향을 문장으로)
4. **가치 카드 3개** (아이콘 + 제목 + 1~2줄):
   - 페이지 문맥 기반 AI — 사용자 컨텍스트를 잃지 않는 AI 파이프라인
   - Action Tools 에이전트 — 브라우저를 직접 조작하는 안전 경계 기반 자동화
   - 스크립트 실행/등록 인프라 — 사이트별 확장 자산을 안전하게 운영
5. **B2B CTA**
   - Primary: "파트너십 문의" → mailto 링크 (1차)
   - Secondary: "기술 문서 보기" → 문서 링크 (1차엔 placeholder 가능)

#### 피해야 할 표현

- 특정 고객사를 암시하는 표현 (레퍼런스 확정 전)
- "엔터프라이즈 급" 과장 (현재 단계와 괴리)
- 일반 제품 카피와 구분되지 않는 무색 표현 (이 섹션이 왜 따로 있는지 흐려짐)

#### 1차 구현 범위

- 문의 수신은 실제 폼이 아닌 **mailto 링크** 또는 **단일 문의 채널 URL**로 단순화
- 실제 고객사/로고/사례는 **없으면 과감히 생략** (placeholder 아이콘만)
- 구현 상태 배지는 이 섹션에 **사용하지 않는다** (기술 성숙도 주장이 아니라 **적용 가능성 제안**이 목적)

## 7. 콘텐츠 표현 가이드

### 7.1 문장 톤

- 소개형, 제품형 문장
- 지나치게 기술 문서처럼 쓰지 않기
- 과장된 마케팅 문구보다 신뢰감 있는 표현 우선

### 7.2 표현 방식

좋은 방향:

- “현재는 …가 가능하다”
- “진행 중인 방향은 …이다”
- “민감한 작업은 보수적으로 처리한다”

피해야 할 방향:

- “모든 웹사이트를 완전히 자동화한다”
- “어떤 업무든 알아서 처리한다”
- “무조건 안전하다”

### 7.3 구현 상태 표기 규칙

랜딩 페이지에서는 기능 상태를 세 단계로 나눈다.

- `구현됨`
- `보강 중`
- `계획/검토 중`

이 구분이 없으면 README/기획 문서와 충돌이 생긴다.

## 8. 디자인 방향

### 8.1 톤앤매너

- 생산성 도구 + AI 코파일럿 느낌
- 기술적이되 차갑지 않은 인상
- 문서보다 제품 랜딩 페이지에 가까운 구조

### 8.2 시각 요소 제안

- Hero 영역: 브라우저 패널 + 페이지 + 액션 흐름을 상징하는 구성
- 기능 카드: 아이콘 + 짧은 설명
- 시나리오 섹션: 단계 카드 또는 타임라인
- 확장 방향: 로드맵 카드

### 8.3 반응형 고려

- 데스크톱 우선
- 태블릿/모바일에서도 카드와 문단이 자연스럽게 재배치
- 긴 표는 카드형 또는 stack 형태로 내려오게 설계

## 11. 구현 전 체크리스트

- 소개서의 메시지가 섹션별로 과도하게 중복되지 않는가
- 현재 구현과 기획 상태가 혼동되지 않는가
- Hero에서 제품 정체성이 5초 안에 이해되는가
- 기능 카드가 단순 나열이 아니라 가치 중심으로 쓰였는가
- 안전 원칙이 빠지지 않았는가
- Floating Helper / Studio / Session Script Continuity가 “미래 방향”으로 명확히 구분되는가
- **비즈니스 적용 섹션(§5.10)이 일반 사용자 카피와 혼동되지 않게 분리되어 있는가** *(대상·톤·CTA가 명확히 B2B로 구분)*
- **마무리 CTA(§5.11)와 비즈니스 CTA(§5.10)가 서로 약화시키지 않는가** *(전자는 설치, 후자는 파트너십 문의로 분리)*
